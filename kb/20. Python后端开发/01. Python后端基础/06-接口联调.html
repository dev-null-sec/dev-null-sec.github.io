接口联调 ​

接口联调处理的是“后端接口写出来以后，调用方能不能按约定稳定使用”。调用方可以是前端页面、脚本、CI、另一个后端服务，也可以只是 curl。联调重点不在页面效果，而在接口契约、状态码、错误格式、CORS、文件上传和请求排查。

FastAPI 的 /docs 和 OpenAPI JSON 已经提供了基础契约。接口越清楚，联调时越少靠猜。

一、接口契约 ​

一个接口至少要写清楚几件事：

项示例
方法GET、POST
路径/articles/{article_id}
请求参数path、query、body、header
响应结构成功返回什么字段
状态码200、201、404、422
认证是否需要 Bearer Token

FastAPI 里，这些信息来自路由函数：

python
@router.get(
    "/articles/{article_id}",
    response_model=schemas.ArticleRead,
    responses={404: {"description": "Article not found"}},
)
def get_article(article_id: int):
    ...

response_model 描述成功响应，responses 补充额外状态码说明。文档里能看到这些信息，调用方不需要打开源码猜字段。

二、统一错误格式 ​

FastAPI 默认错误结构已经比较清楚。业务接口里也可以保持一种简单格式：

json
{
  "error": {
    "code": "article_not_found",
    "message": "article not found"
  }
}

新增文件：app/errors.py

python
from pydantic import BaseModel


class ErrorDetail(BaseModel):
    code: str
    message: str


class ErrorResponse(BaseModel):
    error: ErrorDetail


def error_response(code: str, message: str) -> dict:
    return {
        "error": {
            "code": code,
            "message": message,
        }
    }

接口中使用：

python
from fastapi import HTTPException

from app.errors import error_response


if article is None:
    raise HTTPException(
        status_code=404,
        detail=error_response("article_not_found", "article not found"),
    )

如果项目决定统一错误结构，业务错误就要尽量保持一致。否则调用方一会儿解析 detail 字符串，一会儿解析 error.code，联调会变得很散。

三、CORS ​

浏览器里的前端页面跨域调用 API 时，会受到 CORS 限制。后端需要明确允许哪些来源访问。

修改文件：app/main.py

下面只展示 CORS 相关接入，原有 include_router() 保留。

python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware


app = FastAPI(title="FastAPI Basic")

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://127.0.0.1:5173", "http://localhost:5173"],
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "DELETE"],
    allow_headers=["*"],
)

字段含义：

字段含义
allow_origins允许哪些前端来源
allow_credentials是否允许携带 Cookie、认证信息
allow_methods允许哪些 HTTP 方法
allow_headers允许哪些请求头

开发环境可以放本地前端地址。部署环境里，["*"] 不适合和凭据一起使用，来源范围要收窄到真实域名。

四、文件上传 ​

文件上传需要 python-multipart。

bash
uv add python-multipart

新增文件：app/routers/files.py

python
from pathlib import Path
from uuid import uuid4

from fastapi import APIRouter, File, UploadFile


router = APIRouter(prefix="/files", tags=["files"])

UPLOAD_DIR = Path("uploads")


@router.post("")
async def upload_file(file: UploadFile = File(...)):
    UPLOAD_DIR.mkdir(parents=True, exist_ok=True)

    suffix = Path(file.filename or "").suffix
    save_path = UPLOAD_DIR / f"{uuid4().hex}{suffix}"

    content = await file.read()
    save_path.write_bytes(content)

    return {
        "filename": file.filename,
        "saved_as": str(save_path),
        "size": len(content),
    }

UploadFile.filename 来自客户端，不适合直接当保存路径。上面用 uuid4() 生成服务端文件名，只保留后缀。

文件路由也要接到应用入口里：

python
from app.routers import files


app.include_router(files.router)

请求：

bash
curl -X POST http://127.0.0.1:8000/files \
  -F "file=@README.md"

大文件不适合一次 await file.read() 全部读进内存。这里记录最小结构；处理真实大文件时，要分块写入，并限制文件大小和类型。

五、前端调用 ​

浏览器里用 fetch 调用 JSON 接口：

js
async function loadArticles() {
  const response = await fetch("http://127.0.0.1:8000/articles");

  if (!response.ok) {
    throw new Error(`request failed: ${response.status}`);
  }

  return await response.json();
}

带 token：

js
async function loadMe(token) {
  const response = await fetch("http://127.0.0.1:8000/me", {
    headers: {
      Authorization: `Bearer ${token}`,
    },
  });

  return await response.json();
}

前端联调时先看 Network 面板：请求 URL、方法、状态码、Request Payload、Response、Headers。后端日志只看到服务端一侧，浏览器侧的跨域、Header 缺失、请求体格式错误通常要在 Network 里看。

六、curl调试 ​

curl 能排除前端代码影响，直接验证接口。

GET：

bash
curl -i "http://127.0.0.1:8000/articles?page=1&size=20"

POST JSON：

bash
curl -i -X POST http://127.0.0.1:8000/articles \
  -H "Content-Type: application/json" \
  -d '{"title":"FastAPI","content":"接口联调"}'

带 token：

bash
curl -i http://127.0.0.1:8000/me \
  -H "Authorization: Bearer <access_token>"

-i 会显示响应头，状态码、CORS Header、Content-Type 都能一起看到。

七、OpenAPI导出 ​

FastAPI 的 OpenAPI JSON 地址：

text
http://127.0.0.1:8000/openapi.json

保存：

bash
curl http://127.0.0.1:8000/openapi.json -o openapi.json

OpenAPI 可以给前端生成类型、给接口测试工具导入集合，也可以作为接口变更评审材料。接口字段、状态码和认证方式变化时，OpenAPI JSON 也会变化。

八、联调清单 ​

接口联调常见检查点：

层检查点
路由方法和路径是否匹配
参数path/query/body/header 是否按文档传入
认证Token 是否存在，格式是否是 Bearer
CORS浏览器来源是否在 allow_origins 里
状态码401/403/404/422/500 是否能区分
错误格式调用方是否能稳定解析错误
文件上传字段名、大小、类型、保存路径
日志请求 ID、用户、路径、错误原因

接口本身通过 /docs 或 curl 验证后，再接前端页面。接口本身不通时，前端页面里的状态管理和渲染逻辑只会增加排查变量。

九、常见问题 ​

现象常见原因排查入口
浏览器报 CORS后端没允许当前 origin，或预检请求失败Network、CORS 配置
curl 正常但前端失败Header、请求体、跨域或前端 URL 不一致Network
返回 422请求体字段缺失、类型不对、Content-Type 错误响应 detail
返回 401没带 token、token 过期、签名错误Authorization Header
返回 403当前用户角色不允许权限依赖
上传文件为空表单字段名不是 file，或没用 multipartRequest Payload
文档和实际不一致路由函数缺 response_model，错误返回没记录/docs、源码

联调问题通常出在两端约定不一致。把方法、路径、参数、状态码、错误格式和认证方式固定下来，前端页面和自动化测试才有稳定依据。
