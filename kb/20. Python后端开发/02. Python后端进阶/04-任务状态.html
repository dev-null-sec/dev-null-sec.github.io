任务状态 ​

后台任务只返回 queued 还不够。调用方通常还需要知道任务是否开始、是否完成、失败原因是什么、能不能重试、是否已经超时。任务状态就是把这些信息保存下来。

任务状态先落在数据库表里。轻量场景未必马上上 Celery 或消息队列，但状态模型要先设计清楚。

一、状态流转 ​

常见状态：

状态含义
queued已创建，等待执行
running正在执行
succeeded执行成功
failed执行失败
canceled已取消
timeout执行超时

状态流转大致是：

不是所有任务都支持取消。取消通常表示“系统不再继续处理这个任务”，已经发出去的外部请求未必能撤回。

二、任务表 ​

SQLAlchemy 模型示例：

新增文件：app/models/task.py

python
from datetime import datetime

from sqlalchemy import DateTime, Integer, String, Text
from sqlalchemy.orm import Mapped, mapped_column

from app.database import Base


class TaskRecord(Base):
    __tablename__ = "task_records"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    name: Mapped[str] = mapped_column(String(100), nullable=False)
    status: Mapped[str] = mapped_column(String(20), nullable=False, default="queued")
    message: Mapped[str | None] = mapped_column(Text)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, nullable=False)
    started_at: Mapped[datetime | None] = mapped_column(DateTime)
    finished_at: Mapped[datetime | None] = mapped_column(DateTime)

message 记录失败原因或简短结果。更详细的日志可以放到单独任务日志表，避免状态表越来越大。

三、创建任务 ​

接口创建任务时，先写入数据库，再把任务 ID 交给后台函数。

修改文件：app/routers/tasks.py

python
from fastapi import BackgroundTasks


@router.post("/reports")
def create_report(background_tasks: BackgroundTasks, db: Session = Depends(get_db)):
    task = TaskRecord(name="generate_report", status="queued")
    db.add(task)
    db.commit()
    db.refresh(task)

    background_tasks.add_task(run_report_task, task.id)
    return {"task_id": task.id, "status": task.status}

后台任务只接收 task.id。任务函数内部重新创建数据库 Session，再读取任务记录。

四、执行状态 ​

任务函数里更新状态：

新增文件：app/services/tasks.py

这里的 SessionLocal 来自 app.database，TaskRecord 来自任务表模型。后台任务不接收请求里的 db，而是在函数内部重新创建 Session。

python
from datetime import datetime


def run_report_task(task_id: int):
    db = SessionLocal()
    try:
        task = db.get(TaskRecord, task_id)
        if task is None:
            return

        task.status = "running"
        task.started_at = datetime.utcnow()
        db.commit()

        build_report(task_id)

        task.status = "succeeded"
        task.message = "report generated"
        task.finished_at = datetime.utcnow()
        db.commit()
    except Exception as exc:
        task.status = "failed"
        task.message = str(exc)
        task.finished_at = datetime.utcnow()
        db.commit()
    finally:
        db.close()

任务里不要复用请求里的 Session。请求结束后，依赖函数会关闭 Session，后台任务继续使用它容易出错。

五、查询任务 ​

状态查询接口：

python
@router.get("/tasks/{task_id}")
def get_task(task_id: int, db: Session = Depends(get_db)):
    task = db.get(TaskRecord, task_id)
    if task is None:
        raise HTTPException(status_code=404, detail="task not found")

    return {
        "id": task.id,
        "name": task.name,
        "status": task.status,
        "message": task.message,
        "created_at": task.created_at,
        "started_at": task.started_at,
        "finished_at": task.finished_at,
    }

调用方拿到 task_id 后轮询这个接口。任务量大时，轮询频率要控制，避免状态查询本身变成数据库压力。

六、失败重试 ​

重试可以创建新任务，也可以复用旧任务增加 retry_count。基础模型里加几个字段：

python
retry_count: Mapped[int] = mapped_column(Integer, default=0, nullable=False)
max_retries: Mapped[int] = mapped_column(Integer, default=3, nullable=False)

重试逻辑：

python
def mark_failed_or_retry(db: Session, task: TaskRecord, message: str):
    if task.retry_count < task.max_retries:
        task.retry_count += 1
        task.status = "queued"
        task.message = f"retry scheduled: {message}"
    else:
        task.status = "failed"
        task.message = message

    db.commit()

重试要有上限。没有上限的失败任务会一直占用执行资源，外部系统故障时尤其明显。

七、超时 ​

超时可以放在任务执行函数里，也可以由调度器扫描。

python
from datetime import datetime, timedelta
from sqlalchemy import select


def mark_timeout_tasks(db: Session):
    deadline = datetime.utcnow() - timedelta(minutes=10)
    tasks = db.scalars(
        select(TaskRecord).where(
            TaskRecord.status == "running",
            TaskRecord.started_at < deadline,
        )
    ).all()

    for task in tasks:
        task.status = "timeout"
        task.message = "task exceeded 10 minutes"
        task.finished_at = datetime.utcnow()

    db.commit()

这种扫描适合处理“进程中途退出，任务停在 running”的情况。否则任务表里会一直留着不结束的运行状态。

八、取消 ​

取消接口通常只改状态：

python
@router.post("/tasks/{task_id}/cancel")
def cancel_task(task_id: int, db: Session = Depends(get_db)):
    task = db.get(TaskRecord, task_id)
    if task is None:
        raise HTTPException(status_code=404, detail="task not found")

    if task.status not in {"queued", "running"}:
        raise HTTPException(status_code=409, detail="task already finished")

    task.status = "canceled"
    task.message = "canceled by user"
    task.finished_at = datetime.utcnow()
    db.commit()
    return {"id": task.id, "status": task.status}

运行中的任务函数要定期读取状态，发现已经 canceled 时主动退出。否则取消只会改变数据库状态，任务本身仍然继续跑。

九、任务日志 ​

状态表只存最终状态，过程日志可以单独一张表：

python
class TaskLog(Base):
    __tablename__ = "task_logs"

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    task_id: Mapped[int] = mapped_column(Integer, nullable=False)
    level: Mapped[str] = mapped_column(String(20), nullable=False)
    message: Mapped[str] = mapped_column(Text, nullable=False)
    created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow, nullable=False)

日志接口：

python
@router.get("/tasks/{task_id}/logs")
def list_task_logs(task_id: int, db: Session = Depends(get_db)):
    statement = (
        select(TaskLog)
        .where(TaskLog.task_id == task_id)
        .order_by(TaskLog.id.asc())
    )
    return db.scalars(statement).all()

任务排查时，状态告诉“现在是什么结果”，日志告诉“过程发生了什么”。

十、常见问题 ​

现象常见原因排查入口
任务一直 running进程退出、异常没处理、没有超时扫描任务表、应用日志
状态成功但没结果任务状态更新太早，结果写入失败任务函数顺序
重试不停没有 retry 上限retry_count
取消无效任务函数不检查 canceled 状态执行循环
查询状态很慢任务表没分页、日志表太大索引、分页

任务状态是后台任务的外部可见部分。没有状态表时，后台任务失败只能去翻日志；有状态和日志后，接口调用方可以自己判断任务进度和失败原因。
