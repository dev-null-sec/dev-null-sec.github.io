API设计与REST ​

后端接口数量少的时候，路径写成 /getArticle、/createArticle 也能跑。接口一多，前端、脚本和测试工具就要反复猜：列表接口叫什么，创建接口用什么方法，失败时看哪个字段，重复请求会不会产生两条数据。

REST 风格把接口当成资源的访问约定。文章、用户、文件、任务、事件都近似看作资源，HTTP 方法表达对资源的动作，状态码和响应结构表达结果。接口设计先固定路径、方法、状态码、分页、筛选和幂等性这些基础约定。

一、资源路径 ​

资源路径通常用名词，不用动词。

动作路径方法
列表/articlesGET
详情/articles/{article_id}GET
创建/articlesPOST
整体更新/articles/{article_id}PUT
局部更新/articles/{article_id}PATCH
删除/articles/{article_id}DELETE

动词放在 HTTP 方法里。GET /articles/1 比 GET /getArticle?id=1 更容易和文档、客户端生成工具、网关规则对齐。

特殊动作可以放在资源子路径里：

text
POST /articles/{article_id}/publish
POST /tasks/{task_id}/cancel
POST /files/{file_id}/download-url

这些动作不是普通 CRUD。路径里保留动作名，能让接口含义更清楚。

二、HTTP方法 ​

常见方法：

方法含义是否常见幂等
GET读取是
POST创建或提交动作否
PUT整体替换是
PATCH局部更新通常按实现决定
DELETE删除通常是

幂等表示同一个请求执行一次和执行多次，最终结果一致。PUT /articles/1 每次把文章更新成同一份内容，结果一致；POST /articles 每次都可能创建新文章，结果不一致。

幂等性会影响重试。客户端、网关或任务系统遇到超时时，幂等接口更容易安全重试。

三、状态码 ​

状态码要表达接口当前发生了什么。

状态码场景
200读取、更新成功并返回内容
201创建成功
204删除成功且无响应体
400请求语义错误
401未认证
403已认证但无权限
404资源不存在
409资源冲突
422请求参数或请求体校验失败
500服务内部错误

FastAPI 里设置创建状态码：

python
from fastapi import APIRouter, status


router = APIRouter(prefix="/articles", tags=["articles"])


@router.post("", status_code=status.HTTP_201_CREATED)
def create_article():
    return {"id": 1}

删除接口返回 204：

python
@router.delete("/{article_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_article(article_id: int):
    return None

204 不带响应体。客户端看到 204 时，通常只判断删除成功，不继续解析 JSON。

四、分页 ​

列表接口需要分页。最常见的是 page + size 或 offset + limit。

python
from typing import Annotated

from fastapi import Query


@router.get("")
def list_articles(
    page: Annotated[int, Query(ge=1)] = 1,
    size: Annotated[int, Query(ge=1, le=100)] = 20,
):
    return {"page": page, "size": size, "items": []}

响应里可以带分页信息：

json
{
  "items": [],
  "page": 1,
  "size": 20,
  "total": 0
}

size 不设上限时，一个请求可能要求返回大量数据。列表接口的默认值和上限要在入口处固定。

五、筛选和排序 ​

筛选参数通常放在查询字符串里：

text
GET /articles?keyword=fastapi&published=true&page=1&size=20

排序也可以放查询参数：

text
GET /articles?sort=-created_at

-created_at 表示按创建时间倒序，created_at 表示正序。这种约定要在接口文档里写清楚，不能让调用方靠猜。

FastAPI 示例：

python
@router.get("")
def list_articles(
    keyword: str | None = None,
    published: bool | None = None,
    sort: str = "-created_at",
):
    return {
        "keyword": keyword,
        "published": published,
        "sort": sort,
    }

published: bool | None = None 能区分“不筛选”和“筛选 false”。如果默认写成 False，调用方不传参数时也会只查未发布数据。

六、错误格式 ​

业务错误保持稳定结构：

json
{
  "error": {
    "code": "article_not_found",
    "message": "article not found"
  }
}

字段含义：

字段含义
code稳定错误码，便于前端和脚本判断
message给人看的错误说明

错误码比错误文案更稳定。文案可能因为语言和描述调整变化，错误码应该尽量少变。

七、版本 ​

接口版本常见两种写法：

text
/api/v1/articles

或者用域名、Header 做版本。轻量后端里，路径版本最直观。

python
api_v1 = APIRouter(prefix="/api/v1")
api_v1.include_router(articles.router)
app.include_router(api_v1)

版本号不是每次改字段都增加。通常是兼容性被破坏时才开新版本，比如删除字段、改字段含义、改响应结构。

八、常见问题 ​

现象常见原因排查入口
路径越来越乱动词和名词混用，资源边界不清路由表
列表接口很慢没分页、筛选条件没索引SQL、慢查询
前端解析错误响应结构不稳定response_model、OpenAPI
重试导致重复数据非幂等接口被重复调用HTTP 方法、业务唯一键
错误难处理状态码和错误结构混乱错误返回约定

接口设计最后落到调用方的稳定判断：请求该怎么发，成功是什么样，失败是什么样，重复调用会发生什么。
