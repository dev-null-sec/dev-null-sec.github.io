域名和 DNS ​

网站最后要连到 IP 地址，但人很难记住一串数字。域名就是给 IP 起一个容易记的名字，DNS 负责把这个名字查回地址。

平时访问 www.example.com，浏览器不会凭空知道它在哪里。它要先问 DNS：这个名字对应哪个 IP。网站迁移、接入 CDN、切换机房、内网服务发现，很多事情都和 DNS 有关。

一、域名长什么样 ​

域名从右往左看层级。www.example.com 里，com 是顶级域，example.com 是主域名，www 是这个主域名下面的一个名字。

常见命名：

域名常见用途
www.example.com网站首页
api.example.com后端接口
static.example.com静态资源
grafana.example.com监控平台
harbor.example.com镜像仓库

一个主域名下面可以挂很多业务名。运维看到这些名字时，先分清它们分别指向哪个入口：有的是公网入口，有的是内网入口，有的是 CDN，有的是 Kubernetes Ingress。

二、谁来回答域名 ​

客户端一般不会自己从根 DNS 一路查到底。系统里配置的 DNS 服务器会帮它查询，这类服务器常叫递归 DNS。真正保存某个域名解析记录的地方，叫权威 DNS。

日常看到的 8.8.8.8、114.114.114.114、运营商 DNS、公司内网 DNS，多数是递归 DNS。域名托管平台里配置的解析记录，最终由权威 DNS 对外回答。

三、解析记录 ​

DNS 记录不是只有 IP。不同记录做不同事情。

类型作用示例
A域名指向 IPv4www -> 203.0.113.10
AAAA域名指向 IPv6www -> 2001:db8::10
CNAME域名指向另一个域名www -> cdn.example.net
MX邮件服务器example.com -> mail.example.com
TXT文本验证、SPF、DKIM 等v=spf1 ...
NS指定权威 DNSexample.com -> ns1.example.net

网站入口最常见的是 A、AAAA 和 CNAME。接 CDN 时，业务域名常 CNAME 到 CDN 分配的域名，再由 CDN 返回边缘节点 IP。

四、TTL 和缓存 ​

DNS 查询结果会被缓存一段时间。TTL 就是缓存时间。

TTL 写 300，表示结果可以缓存 300 秒。TTL 写得长，查询少，切换慢；TTL 写得短，切换快一些，查询会多一些。

域名迁移前，经常会把 TTL 调低。比如原来是 3600 秒，先改成 300 秒，等旧缓存逐渐过期后再改 A 记录。这样切错了也能更快回到旧地址。

已经被缓存的旧结果不会因为修改解析记录立刻消失。某些浏览器、操作系统、运营商 DNS 还会有自己的缓存，所以“解析已经改了，但还有人访问旧 IP”在迁移时很常见。

五、内网名字 ​

DNS 不只用于公网。公司内部、云 VPC、Kubernetes 集群里也会用自己的域名。

场景常见名字
公司内部平台gitlab.company.local
云 VPC 内服务mysql.internal.example.com
Kubernetes Serviceapi.default.svc.cluster.local

内网域名让服务不用写死 IP。数据库切主、Pod 重建、服务扩容以后，IP 可能变化，名字可以保持不变。

Kubernetes 里访问 api.default.svc.cluster.local，本质也是 DNS 查询。Service 名或 namespace 写错时，应用日志里可能出现 lookup api.default.svc.cluster.local: no such host。如果 Service 存在但后端 Pod 没有就绪，DNS 可能解析成功，实际请求会卡在 Service 或 Endpoints 这一层。

六、常见解析异常 ​

DNS 问题不要只看“能不能解析”，还要看解析出了什么。

结果常见含义
NXDOMAIN域名不存在，或者没有对应记录
SERVFAILDNS 服务器处理失败，上游或权威 DNS 可能异常
解析到旧 IP缓存还没过期，或者某处写了固定 hosts
不同地区结果不同CDN、智能解析、运营商缓存差异

NXDOMAIN 不一定就是故障。有些应用会主动尝试多个候选域名，某些候选域名不存在也会打出 NXDOMAIN。多个核心域名同时 SERVFAIL，或者解析耗时突然从几毫秒涨到几秒，更像是 DNS 链路出了问题。

DNS 是访问链路里的名字层。名字指向哪里，用户就从哪里进入系统。
